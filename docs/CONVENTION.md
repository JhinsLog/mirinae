# 🌌 '미리내(mirinae)' 프로젝트 가이드라인

본 문서는 프로젝트의 일관성 있는 구조 증명, 모노레포 관리 효율화, 그리고 DDD/TDD 방법론 기반의 고품질 소프트웨어 구현을 위한 규칙을 정의합니다.

---

## 🌳 1. 브랜치 관리 전략 (Git Flow)

프로젝트의 안정적인 아키텍처 검증을 위해 오직 기능과 모듈을 기준으로 브랜치를 분기하는 Git Flow 전략을 채택합니다.

### 📌 브랜치 생성 규칙
- `[개념]/[모듈]-[기능명]` 서식을 준수합니다.
- 예시: `feature/backend-spatial-query`, `feature/frontend-canvas-optimization`, `fix/backend-redis-timeout`

### 📊 브랜치 생명주기 및 파이프라인 매핑
| 브랜치 종류 | 설명 | 관리 및 병합 규칙 | 연동 CI/CD 파이프라인 |
| :--- | :--- | :--- | :--- |
| **main** | 제품 배포용 최종 생산 코드 | 직접 Push 절대 금지. `develop`에서 검증 완료된 내용만 PR을 통해 병합 가능. | **CD 작동**: Rocky Linux 운영 서버로 무중단 자동 배포 프로세스 가동 |
| **develop** | 차기 출시 버전 통합 브랜치 | 모든 기능 개발 브랜치의 종착지이며, 통합 테스트가 수행되는 공간입니다. | **CI 작동**: 변경 사항 발생 시 빌드 및 단위 테스트 자동 수행 및 검증 |
| **feature/** | 신규 기능 개발 브랜치 | `develop`에서 분기하며, 작업 완료 후 `develop`으로 Pull Request를 생성합니다. | **로컬 검증**: 푸시 전 로컬 빌드 및 소스 코드 정적 분석(Lint) 통과 필수 |
| **hotfix/** | 배포 버전 긴급 버그 수정 | `main`에서 직접 분기하며, 수정 완료 후 `main`과 `develop` 양측에 모두 병합합니다. | **CI/CD 가동**: 병합 즉시 패치 버전 빌드 및 롤링 배포 수행 |

---

## ✍️ 2. 커밋 메시지 규칙 (Monorepo Conventional Commits)

모노레포 환경에서 백엔드와 프론트엔드의 변경 이력을 명확히 격리하고 추적하기 위해 **Type(Scope): Subject** 구조를 엄수합니다. 별도의 이슈 번호 연동 없이 본문(Body) 중심의 직관적인 기록을 지향합니다.

### 📋 커밋 형태 (Format)
```text
<Type>(<Scope>): <Subject>

[선택적 Body: 변경 이유, DDD 설계 변경 사항 또는 TDD 테스트 관점 기술]
